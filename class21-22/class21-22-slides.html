Authentication vs Authorisation — Two Different Things

🧠 In Plain English

Authentication = Proving WHO you are.
"I am Tolu. Here is my password."

Authorisation = Deciding WHAT you can do.
"Tolu is allowed to see their own dashboard but NOT the admin panel."

You must be authenticated before you can be authorised.

Real-Life Analogy

🪪

Authentication

Showing your ID at the door. Proves you are who you say you are.

🎟️

Authorisation

The VIP wristband that says what areas you can access inside.

In Class 21: We focus on Authentication — the login flow, storing the proof of identity, and protecting routes. Authorisation (roles, permissions) is a more advanced topic for the Capstone.

JWT Tokens — Your Proof of Identity

🧠 In Plain English

When you log in, the server checks your email and password. If they're correct, it stamps a "ticket" (a JWT token) and gives it to you. Every time you want to do something that requires being logged in, you show that ticket.

The server doesn't need to check your password again — it just verifies the ticket is genuine. Faster and more scalable.

↓
5. Every future request includes the token

Auth State with Context — The React Pattern

🧠 In Plain English

Your Navbar needs to know if the user is logged in (to show "Log Out" vs "Log In"). Your Dashboard needs it (to show the right user's data). Your ProtectedRoute needs it. If you stored auth state in one component, you'd have to pass it as props everywhere — prop drilling nightmare.
Instead: store auth state in a Context that any component can read from anywhere in the tree. One source of truth, accessible everywhere.

// src/context/AuthContext.jsx
import { createContext, useContext, useState } from 'react';

const AuthContext = createContext(null);

// This wraps the whole app in main.jsx
export const AuthProvider = ({ children }) => {
  // Try to restore session from localStorage on load
  const [user, setUser] = useState(() => {
    const saved = localStorage.getItem('user');
    return saved ? JSON.parse(saved) : null;
  });

  const login = (userData) => {
    setUser(userData);
    localStorage.setItem('user', JSON.stringify(userData));
  };

  const logout = () => {
    setUser(null);
    localStorage.removeItem('user');
  };

  return (
    <AuthContext.Provider value={{ user, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
};

// Custom hook — any component calls this to get auth state
export const useAuth = () => useContext(AuthContext);
// main.jsx — wrap App with AuthProvider
import { AuthProvider } from './context/AuthContext';

ReactDOM.createRoot(...).render(
  <BrowserRouter>
    <AuthProvider>    ← add this
      <App />
    </AuthProvider>
  </BrowserRouter>
);
// Any component — just call useAuth()
import { useAuth } from '../context/AuthContext';

const Navbar = () => {
  const { user, logout } = useAuth();

  return (
    <nav>
      {user
        ? <button onClick={logout}>Log Out</button>
        : <Link to="/login">Log In</Link>
      }
    </nav>
  );
};

useAuth() is available in ANY component. No props needed. This is the power of Context.

The Login Page — React Hook Form + Auth Context

Live Coding 
// pages/LoginPage.jsx
import { useForm } from 'react-hook-form';
import { useNavigate } from 'react-router-dom';
import { useAuth } from '../context/AuthContext';

const LoginPage = () => {
  const { login } = useAuth();
  const navigate = useNavigate();

  const { register, handleSubmit,
    formState: { errors, isSubmitting },
    setError } = useForm();

  const onSubmit = async (data) => {
    try {
      // Using reqres.in as a free mock auth API
      const res = await fetch(
        'https://reqres.in/api/login', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(data),
      });

      if (!res.ok) throw new Error('Invalid credentials');

      const { token } = await res.json();

      // Store user + token via Auth Context
      login({ email: data.email, token });

      // Redirect to the page they were trying to visit
      navigate('/dashboard');

    } catch {
      setError('root', { message: 'Invalid email or password' });
    }
  };

  return (
    <div className="min-h-screen flex items-center justify-center">
      <form onSubmit={handleSubmit(onSubmit)}
        className="bg-white p-8 rounded-xl shadow-lg w-full max-w-sm space-y-4">
        <h1 className="text-2xl font-bold">Log In</h1>

        {/* root error from failed API call */}
        {errors.root &&
          <p className="text-red-500 text-sm bg-red-50 p-2 rounded">
            {errors.root.message}
          </p>}

        {/* email + password inputs with register() */}
        <input type="email" placeholder="Email"
          className="w-full border p-2 rounded"
          {...register('email', { required: 'Required' })} />

        <input type="password" placeholder="Password"
          className="w-full border p-2 rounded"
          {...register('password', { required: 'Required' })} />

        <button type="submit" disabled={isSubmitting}
          className="w-full bg-indigo-600 text-white py-2 rounded">
          {isSubmitting ? 'Logging in...' : 'Log In'}
        </button>
      </form>
    </div>
  );
};

Protected Routes — Block Unauthenticated Users

🧠 In Plain English

Right now, anyone can type /dashboard in the URL bar and reach it. A protected route is a wrapper that checks: "Is this person logged in?" If yes → show the page. If no → redirect to /login automatically.

// components/ProtectedRoute.jsx
import { Navigate } from 'react-router-dom';
import { useAuth }  from '../context/AuthContext';

const ProtectedRoute = ({ children }) => {
  const { user } = useAuth();

  // Not logged in → kick to login page
  if (!user) return <Navigate to="/login" replace />;

  // Logged in → show whatever is wrapped inside
  return children;
};

export default ProtectedRoute;
// App.jsx — wrap protected pages
<Routes>
  <Route path="/"         element={<HomePage/>} />
  <Route path="/login"    element={<LoginPage/>} />

  {/* Only logged-in users can reach /dashboard */}
  <Route path="/dashboard"
    element={
      <ProtectedRoute>
        <Dashboard />
      </ProtectedRoute>
    }
  />
</Routes>

What Happens in Each Scenario

Logged in → visits /dashboard

user exists in Context → children rendered → Dashboard shows ✅

Not logged in → visits /dashboard

user is null → Navigate to="/login" → redirected ✅

Refreshes the page (logged in)

Context re-reads localStorage → user restored → Dashboard stays ✅

replace on Navigate means the login page replaces the current history entry. So after logging in, pressing "Back" won't return them to the login page — they'll go further back instead. Better UX.

  • ✅ AuthContext.jsx — user, login(), logout()
  • ✅ AuthProvider wraps app in main.jsx
  • ✅ useAuth() hook used in any component
  • ✅ LoginPage calls login() + navigate()

⚠️ localStorage Security Note

For production apps, tokens in localStorage are vulnerable to XSS attacks. Secure alternatives include HTTP-only cookies. For learning and this course, localStorage is fine. Just know the limitation exists.

Environment Variables — Hide Your Secrets

🧠 In Plain English

An environment variable is a value stored OUTSIDE your code, in a special file that never gets pushed to GitHub. Your code reads the value from this file at runtime. Anyone looking at your GitHub sees only import.meta.env.VITE_API_KEY — never the actual key.

# .env  ← this file NEVER goes to GitHub
VITE_TMDB_KEY=abc123yourapikey
VITE_APP_TITLE=My Movie App

# In your React code (Vite syntax):
const API_KEY = import.meta.env.VITE_TMDB_KEY;

const url = `https://api.themoviedb.org/3/movie/popular
  ?api_key=${API_KEY}`;

# .env.example  ← this DOES go to GitHub
VITE_TMDB_KEY=your_tmdb_api_key_here
VITE_APP_TITLE=My App

The Rules in Vite

  • Variables must start with VITE_ to be accessible in React code
  • Read them with import.meta.env.VITE_YOUR_VAR
  • Restart the dev server after changing .env
  • Add .env to .gitignore — never commit it

⚠️ .gitignore Check

Vite's default .gitignore already ignores .env files. But always double-check: open .gitignore and confirm .env is listed before your first commit with API keys.

🤖 If you accidentally committed your API key: "I accidentally pushed my API key to GitHub. What should I do right now?" (Hint: revoke + regenerate the key immediately.)

Development vs Production — What Changes

🧠 In Plain English

While you're building (npm run dev): your code is uncompressed, has source maps, updates instantly, runs hot-reload. Great for development, terrible for users — it's slow and reveals your source code.

When you ship (npm run build): Vite minifies, compresses, tree-shakes (removes unused code), and bundles everything into optimised static files that load fast.

npm run build

# Run this when you're ready to deploy:
npm run build

# Creates a dist/ folder:
dist/
├── index.html
├── assets/
│   ├── index-a3f4b2c.js   ← minified JS
│   └── index-8c2d1a4.css  ← minified CSS

# Preview the production build locally:
npm run preview

npm run dev (Development)

  • Fast hot-reload as you edit
  • Readable, uncompressed code
  • Detailed error messages
  • Source maps for debugging
  • Console.logs are visible

npm run build (Production)

  • Minified + compressed files
  • Tree-shaking (unused code removed)
  • Smaller bundle = faster load
  • Optimised for real users

Always run npm run build before deploying. Then npm run preview to test the production build locally before it goes live.

Lazy Loading Routes — Load Pages Only When Needed

🧠 In Plain English

Right now, when your app loads, it downloads ALL your pages — even ones the user may never visit. Lazy loading changes this: each page only downloads when the user actually navigates to it.

For a 5-page app: instead of one 500KB bundle, you get five 100KB chunks. The home page loads faster. Other pages load on demand.

// App.jsx BEFORE — all pages downloaded upfront
import HomePage    from './pages/HomePage';
import MoviePage   from './pages/MoviePage';
import SearchPage  from './pages/SearchPage';

// App.jsx AFTER — each page loads on demand
import { lazy, Suspense } from 'react';

const HomePage   = lazy(() => import('./pages/HomePage'));
const MoviePage  = lazy(() => import('./pages/MoviePage'));
const SearchPage = lazy(() => import('./pages/SearchPage'));

// Wrap Routes in Suspense (shows fallback while loading)
<Suspense fallback={<LoadingSpinner/>}>
  <Routes>
    <Route path="/"        element={<HomePage/>}  />
    <Route path="/movie/:id" element={<MoviePage/>} />
  </Routes>
</Suspense>

Without vs With Lazy Loading

❌ Without — one big bundle

500KB downloaded even if user only visits Home

✅ With lazy — small chunks on demand

Home loads 100KB. Other pages load only when visited.

The only change is wrapping your imports in lazy(() => import(...)) and adding <Suspense fallback=...> around your Routes. Zero logic changes needed.

# Copy this file to .env and fill in your values.

VITE_APP_NAME=My Movie App
VITE_API_BASE=https://api.themoviedb.org/3

✅ .gitignore — confirm these are listed

# .gitignore
node_modules
dist
.env
.env.local
.env.*.local

🧠 The .env.example Workflow

When someone clones your project they won't have your .env file — that's the whole point. They look at .env.example to know what variables they need to create. It's the instruction manual for secrets.

Using env vars in your code

// Always store in a constant at the top of the file
const TMDB_KEY = import.meta.env.VITE_TMDB_KEY;
const BASE_URL  = import.meta.env.VITE_API_BASE;

// Then use the constant — not the env var directly
const url = `${BASE_URL}/movie/popular?api_key=${TMDB_KEY}`;

// Guard against missing variables:
if (!TMDB_KEY) {
  console.error('Missing VITE_TMDB_KEY in .env');
}

Remember: VITE_ prefix is mandatory. Variables without it are available during build but NOT in browser-side React code. TMDB_KEY won't work. VITE_TMDB_KEY will.

🧠 Why Auth Is a Great AI Use Case

Auth patterns are highly standardised — Context + localStorage + ProtectedRoute is the same structure in almost every React app. AI has seen thousands of implementations and will give you accurate, complete code.

But: read every line. Auth bugs are security bugs. Understanding what your auth code does is not optional.

⚠️ Security Mistakes AI Can Make

  • Storing sensitive data in state instead of a ref (re-renders expose it)
  • Forgetting replace on the Navigate redirect
  • Checking auth state before Context has loaded
