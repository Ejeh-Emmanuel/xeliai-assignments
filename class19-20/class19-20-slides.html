Tailwind CSS

Stop writing custom CSS files. Style directly in your JSX.

🧠 Before We Start

Right now, styling means: write a CSS file, invent a class name, switch files, write the CSS, switch back, hope you spelled the class name the same in both places. Tailwind removes all of that — you style directly where you're already working.

CSS Approaches
4 / 26

3 Ways to Style React — Quick Recap

🧠 In Plain English

You've already used two of these without thinking about it. Today we add a third — and it's the one most companies actually use in production.

1. Separate CSS File

/* Card.css */
.card {
  padding: 16px;
  border-radius: 8px;
  background: white;
}

// Card.jsx
import './Card.css';
<div className="card">...</div>

Two files to manage. Class names can clash globally.

2. Inline Styles

// No CSS file needed
<div style={{
  padding: '16px',
  borderRadius: '8px',
  background: 'white'
}}>...</div>

No hover/media queries. Gets messy fast with many styles.

3. Tailwind (Today!) 🆕

// One file. Style right where you write JSX.
<div className="p-4 rounded-lg bg-white
  hover:shadow-lg md:p-6">
  ...
</div>

No file switching. Hover & responsive built in. Industry standard.

Why does this matter for your career? Tailwind is used by Netflix, GitHub, OpenAI, and thousands of startups. Job postings frequently list it as a requirement. You're learning an industry-standard tool, not a course gimmick.

What Is Tailwind? — Utility Classes Explained

🧠 In Plain English

Normal CSS: you invent a name like .product-card, then write what it means. Tailwind: the name is what it means. p-4 means "padding: 1rem". text-center means "text-align: center". You're not naming things — you're describing them directly.

Think of it like LEGO bricks. Each utility class is one small brick. You combine many small bricks to build the exact shape you want.

// Traditional CSS approach:
<div className="card">Hello</div>

.card {
  padding: 16px;
  background-color: white;
  border-radius: 8px;
  font-weight: bold;
  text-align: center;
}

// ─────────────────────────────

// Tailwind approach — same result:
<div className="p-4 bg-white rounded-lg
  font-bold text-center">
  Hello
</div>
// No separate CSS file at all!

Reading Tailwind Class Names

p-4 padding: 1rem (16px)
m-2 margin: 0.5rem (8px)
text-center text-align: center
bg-blue-500 background: blue, shade 500
rounded-lg border-radius: large
font-bold font-weight: bold

✅ The Spacing Scale (memorise this)

Tailwind numbers = multiples of 0.25rem (4px). p-1=4px, p-2=8px, p-4=16px, p-8=32px. Once you know this scale, every spacing class makes sense instantly.

🤖 Forgot a class name? Ask AI: "What's the Tailwind class for a 2px solid grey border with rounded corners?" Faster than searching docs while learning.

Installing Tailwind in a Vite Project

Live Setup

Step 1 — Install packages

npm install tailwindcss @tailwindcss/vite

Step 2 — Add the Vite plugin

// vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import tailwindcss from '@tailwindcss/vite';

export default defineConfig({
  plugins: [react(), tailwindcss()],
});

Step 3 — Import Tailwind in your CSS

/* src/index.css — replace everything with: */
@import "tailwindcss";

✅ Checklist

  • index.css is imported in main.jsx
  • Restart your dev server (npm run dev) after config changes
  • Test: add className="text-red-500" to any element. Does it turn red?

⚠️ Common Mistake

Forgetting to restart the dev server after editing vite.config.js. Config changes need a full restart — refreshing the browser isn't enough.

Responsive Design — Breakpoint Prefixes

🧠 In Plain English

Remember media queries from Class 5? @media (min-width: 768px) { ... } — that was a lot of typing for one rule. In Tailwind, you just prefix any class with a breakpoint name. md:flex means "apply flex ONLY on screens 768px and wider." Mobile styles are the default (no prefix); add prefixes for bigger screens.

The Breakpoint Scale

(none) mobile — default, always applies
sm: ≥ 640px — large phones
md: ≥ 768px — tablets
lg: ≥ 1024px — laptops
xl: ≥ 1280px — desktops
// Mobile-first: 1 column → 2 → 4 as screen grows
<div className="grid grid-cols-1
  md:grid-cols-2 lg:grid-cols-4 gap-4">
  <Card /> <Card /> <Card /> <Card />
</div>

// Read it like a sentence:
// "1 column normally, 2 on tablets,
//  4 on laptops and up"

Visual: Same Grid, 3 Screen Sizes

📱 Mobile (default — grid-cols-1)

📱 Tablet (md:grid-cols-2)

💻 Laptop (lg:grid-cols-4)

Same class string. Three layouts. Resize your browser right now while looking at any Tailwind site — watch the grid columns change live.

Hover, Focus & Other States — No JavaScript Needed

🧠 In Plain English

Want a button to get darker when you hover? In plain CSS that's a :hover selector in a separate rule. In Tailwind, you just add hover: in front of any class, right in the same className string.

// A button with hover + focus states
<button className="px-4 py-2 bg-indigo-600
  text-white rounded-lg font-semibold
  hover:bg-indigo-700
  focus:ring-2 focus:ring-indigo-300
  transition-colors">
  Save Changes
</button>

// hover:bg-indigo-700  → darker on hover
// focus:ring-2         → ring when focused
// transition-colors    → smooth animation
Save Changes

Hover over this in your browser — Tailwind buttons feel the same way

Common State Prefixes

hover: when mouse is over the element
focus: when an input/button is focused
disabled: when an input/button is disabled
active: while being clicked

Combining Prefixes

You can stack them: md:hover:bg-blue-700 means "on tablets and up, when hovering, make the background blue-700." Read left to right like layered conditions.

Forms & Validation

Make sure the data your users submit is actually usable.

🧠 Before We Start

You've built controlled inputs since Class 14 — one useState per field, onChange updates it. That works for 1-2 fields. For a form with 5+ fields and validation rules, that approach becomes a LOT of repetitive code. React Hook Form solves this.

Forms Basics
14 / 26

Controlled vs Uncontrolled — Quick Recap

🧠 In Plain English

Controlled (what you've been doing): React state holds the value. Every keystroke updates state, state updates the input. React is "in control."

Uncontrolled: the DOM holds the value itself, like a normal HTML form. You only "read" the value when needed (e.g. on submit) using a ref.

Controlled (Class 14 style)

const [name, setName] = useState('');

<input
  value={name}
  onChange={e => setName(e.target.value)}
/>
// Every keystroke = a re-render
// Fine for 1-2 fields.
// For 6 fields = 6 useState + 6 onChange 😩

Uncontrolled (React Hook Form style)

const { register } = useForm();

<input {...register('name')} />
// register() handles value + onChange for you
// No useState per field needed!
// React Hook Form reads values only
// when you submit — fewer re-renders

✅ Why React Hook Form?

  • No useState for each field
  • Built-in validation rules
  • Automatic error messages
  • Better performance on big forms
  • Industry standard — used everywhere

You're not "forgetting" useState — for forms specifically, this is simply the better tool for the job.

React Hook Form — Basic Setup

Live Coding 
# Install:
npm install react-hook-form
import { useForm } from 'react-hook-form';

const ContactForm = () => {
  const {
    register,         // connects inputs
    handleSubmit,     // wraps your submit fn
    formState: { errors }  // validation errors
  } = useForm();

  const onSubmit = (data) => {
    console.log(data);
    // data = { name: 'Tolu', email: '...' }
    // All fields, automatically collected!
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input {...register('name')}
        placeholder="Your name" />

      <input {...register('email')}
        placeholder="Your email" />

      <button type="submit">Send</button>
    </form>
  );
};

The 3 Things You Get From useForm()

register — spread onto each input
handleSubmit — wraps your submit function
formState.errors — validation errors

🧠 What does {'{...register(\'name\')}'} do?

The {'{...spread}'} drops several props onto the input at once: name, onChange, onBlur, and a ref. It's the same as writing all four manually — register() does it in one line.

handleSubmit(onSubmit) does two things: it calls e.preventDefault() for you (no more forgetting!) AND it only calls your onSubmit if validation passes.

Validation
16 / 26

Validation Rules — Built Right Into register()

🧠 In Plain English

Pass a second argument to register() — an object describing the rules. React Hook Form checks these automatically on submit (and optionally as the user types). No manual if-statements needed.

<input
  {...register('email', {
    required: 'Email is required',
    pattern: {
      value: /^[^\s@]+@[^\s@]+\.[^\s@]+$/,
      message: 'Enter a valid email'
    }
  })}
/>

<input type="password"
  {...register('password', {
    required: 'Password is required',
    minLength: {
      value: 8,
      message: 'Must be at least 8 characters'
    }
  })}
/>

<input type="number"
  {...register('age', {
    required: 'Age is required',
    min: { value: 18, message: 'Must be 18+' }
  })}
/>

Built-in Validation Rules

required — field cannot be empty
minLength / maxLength — text length
min / max — number range
pattern — regex match (email, etc.)

✅ Each rule has a custom message

Instead of a generic "Invalid input", you write the exact message the user sees: "Must be at least 8 characters". This message lives in errors.fieldName.message — shown on the next slide.

Full Example — Styled, Validated Contact Form

Live Coding 
import { useForm } from 'react-hook-form';

const ContactForm = () => {
  const { register, handleSubmit, formState: { errors, isSubmitting } }
    = useForm();

  const onSubmit = async (data) => {
    await new Promise(r => setTimeout(r, 1000)); // fake API call
    console.log('Submitted:', data);
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}
      className="max-w-md mx-auto p-6
        bg-white rounded-xl shadow-md space-y-4">

      <div>
        <label className="block text-sm font-semibold mb-1">
          Name
        </label>
        <input
          className={`w-full p-2 border rounded
            ${errors.name ? 'border-red-500' : 'border-gray-300'}`}
          {...register('name', { required: 'Name is required' })}
        />
        {errors.name &&
          <p className="text-red-500 text-sm mt-1">{errors.name.message}</p>}
      </div>

      <div>
        <label className="block text-sm font-semibold mb-1">Email</label>
        <input
          className={`w-full p-2 border rounded
            ${errors.email ? 'border-red-500' : 'border-gray-300'}`}
          {...register('email', {
            required: 'Email is required',
            pattern: { value: /^[^\s@]+@[^\s@]+\.[^\s@]+$/, message: 'Invalid email' }
          })}
        />
        {errors.email &&
          <p className="text-red-500 text-sm mt-1">{errors.email.message}</p>}
      </div>

      <button type="submit" disabled={isSubmitting}
        className="w-full bg-indigo-600 text-white py-2
          rounded font-semibold hover:bg-indigo-700
          disabled:opacity-50">
        {isSubmitting ? 'Sending...' : 'Send Message'}
      </button>
    </form>
  );
};

Live Preview

Send Message

isSubmitting — built in!

React Hook Form tracks whether your onSubmit async function is still running. Use it to disable the button and show "Sending..." — no extra useState needed.

space-y-4 is a Tailwind trick: it adds vertical margin between direct children automatically. Great for forms — no margin classes needed on each field.
