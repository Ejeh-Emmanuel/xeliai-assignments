Why State?
4 / 26

The Problem — Props Can't Change

❌ Trying to use a regular variable

const Counter = () => {
  let count = 0;

  const handleClick = () => {
    count = count + 1; // ← changes the var
    console.log(count);  // ← logs correctly!
  };

  return (
    <div>
      <p>Count: {count}</p>  // ← always shows 0
      <button onClick={handleClick}>+</button>
    </div>
  );
};

❌ Why it fails: The variable changes in memory, but React has no idea. It doesn't re-render. The screen stays frozen at 0 forever.

✅ Using useState — React knows to re-render

import { useState } from 'react';

const Counter = () => {
  const [count, setCount] = useState(0);

  const handleClick = () => {
    setCount(count + 1); // ← tells React
  };

  return (
    <div>
      <p>Count: {count}</p>  // ← updates! ✅
      <button onClick={handleClick}>+</button>
    </div>
  );
};

✅ Why it works: setCount tells React the value changed. React re-runs the component function and paints the new count to the screen.

What Is State? — A Mental Model

State is a component's memory.

Every time a component renders, local variables start fresh from zero. State is the mechanism React provides to remember values between renders — and to trigger a re-render when those values change.

Real-World Analogies

  • 🛒 Cart count — changes when you add items
  • ❤️ Like button — toggles between liked/not liked
  • 📝 Search box — updates with every keystroke
  • 🌙 Dark mode toggle — switches the whole theme
  • 📋 Form fields — track what the user typed

The Re-render Cycle

1

User triggers an event (click, type, submit)

↓
2

Event handler calls the setter setCount(newValue)

↓
3

React schedules a re-render of this component

↓
4

Component function runs again with new state value

↓
5

Screen updates. User sees the change instantly. ✅

State vs Props
6 / 26

State vs Props — The Architectural Comparison

PROPS  📥  — Passed In From Outside

  • Passed from parent to child
  • Read-only inside the component
  • Component cannot change its own props
  • Different every time the component is reused
  • Like arguments to a function
// Parent passes props:
<UserCard name="Tolu" city="Lagos" />
// Child reads only — never writes:
const UserCard = ({ name, city }) => ...

STATE  🧠  — Lives Inside the Component

  • Declared and owned by the component
  • Mutable — changes via the setter function
  • Private to the component unless lifted up
  • Survives across renders (unlike local variables)
  • Like the component's internal memory
// Component owns its own state:
const [isOpen, setIsOpen] = useState(false);
// Only this component can change it:
setIsOpen(true); // ← triggers re-render

Simple Rule

If data comes from outside → it's a prop. If data lives inside and can change → it's state. You'll often use both in the same component.

The useState Hook — Syntax First Look

// Step 1: Import at the top of the file
import { useState } from 'react';

const MyComponent = () => {

  // Step 2: Declare state with useState()
  const [count, setCount] = useState(0);
  //     ─────  ────────    ─────────  ─
  //     value  setter fn   hook call  initial

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        +1
      </button>
    </div>
  );
};

Breaking It Down — Part by Part

useState(0) — the hook call. 0 is the initial value.
count — the current value. Use it in JSX like any variable.
setCount — the setter. Call it to update the value.
const [a, b] = useState() — array destructuring (from Class 10!).

Naming Convention

Always name the setter set + the state name. count → setCount. isOpen → setIsOpen. username → setUsername. This is the standard across the entire React ecosystem.

useState — Any Data Type Works

The initial value passed to useState() can be any JavaScript type. Match the type to the data you're storing.

// Number — for counts, scores, ages
const [score, setScore] = useState(0);

// String — for names, inputs, labels
const [name, setName] = useState('');

// Boolean — for toggles, flags
const [isOpen, setIsOpen] = useState(false);
const [isLoading, setIsLoading] = useState(false);

// Array — for lists of items
const [items, setItems] = useState([]);

// Object — for related fields
const [user, setUser] = useState({
  name: '',
  email: ''
});

⚠️ null is also valid

// null = "I don't have data yet"
const [user, setUser] = useState(null);

// Conditionally render based on it:
{user && <UserCard name={user.name} />}

✅ Rule of Thumb

Start with the most natural "empty" or "default" value for your data type. 0 for numbers. '' for strings. false for booleans. [] for lists. null when you don't know yet.

One state per concern

Prefer multiple simple state variables over one big object. const [name, setName] = useState('') is cleaner than one massive formData object for simple cases.

Rules of Hooks — Read This Once, Never Forget

Hooks are functions that start with use. React tracks them by call order. Break these rules and your app will behave in bizarre, unpredictable ways.

✅ RULE 1 — Only call hooks at the TOP LEVEL

// ✅ Always at the top, unconditionally:
const MyComp = () => {
  const [count, setCount] = useState(0);
  const [name, setName] = useState('');

  // ❌ NEVER inside an if, loop, or nested fn:
  // if (condition) { const [x] = useState(0); }
  // for (...) { const [y] = useState(0); }
};

✅ RULE 2 — Only call hooks inside React functions

// ✅ Inside a component (works):
const Counter = () => {
  const [n, setN] = useState(0);
};

// ❌ At module level (breaks everything):
// const [n, setN] = useState(0); ← NEVER

⚠️ Why These Rules Exist

React tracks hooks by their position in the call order. If you call useState inside an if, it might run on some renders and not others — breaking React's internal tracking. The hooks must always run in the same order every single render.

ESLint Saves You Here

The eslint-plugin-react-hooks package (included with Vite by default) automatically catches both violations and shows a red underline before you ever run the code. Pay attention to these warnings.

Built-in hooks you'll use: useState (Class 14)  •  useEffect (Class 15)  •  useContext (Class 16)

Live Coding — Your First useState Component

Live Coding 
// src/components/Counter.jsx
import { useState } from 'react';

const Counter = () => {
  const [count, setCount] = useState(0);

  return (
    <div className="counter">
      <h2>Click Counter</h2>

      <p className="count-display">{count}</p>

      <div className="btn-group">
        <button onClick={() => setCount(count - 1)}>−</button>
        <button onClick={() => setCount(0)}>Reset</button>
        <button onClick={() => setCount(count + 1)}>+</button>
      </div>

      {/* Conditional rendering based on state */}
      {count > 10 && <p className="msg">🔥 Over 10!</p>}
      {count < 0  && <p className="msg">❄️ Below zero</p>}
    </div>
  );
};

export default Counter;

What to notice:

  • Three buttons, one shared state variable
  • Each button calls setCount with a different value
  • Reset passes the literal 0, not count - count
  • Conditional messages react to count's value
  • No manual DOM update anywhere

✅ Build this first — then extend:

  • Prevent count going below 0
  • Add a step size prop: +5, +10
  • Change text colour when count is negative

⚡ Shorthand: Inline arrow functions on onClick are fine for simple updates. For complex logic, extract to a named handler function above the return.

How State Triggers Re-renders

The Re-render Contract

When you call setCount(newValue), React does not immediately update count in the current function call. It schedules a new render. On the next render, the component function re-runs and useState returns the new value.

const handleClick = () => {
  setCount(count + 1);
  // count is still the OLD value here!
  console.log(count); // ← logs old value
  // New value only visible on next render
};

⚠️ Common gotcha: Logging state immediately after calling the setter shows the old value. This isn't a bug — state updates are asynchronous. Trust that the re-render will have the correct value.

Functional Updates — The Safe Pattern

// ❌ Can be wrong if updates batch:
setCount(count + 1);
setCount(count + 1); // still uses old count!

// ✅ Functional update — always correct:
setCount(prev => prev + 1);
setCount(prev => prev + 1); // uses latest

When the new state depends on the previous state, always use the functional form: setCount(prev => prev + 1). This guarantees you're working with the most up-to-date value.

✅ For simple counters in class: setCount(count + 1) works fine. Use the functional form whenever the new value depends on the previous value in production apps.

Multiple State Variables — Each Independent

Call useState as many times as you need. Each call creates a completely independent piece of state with its own value and setter.

// src/components/ProfileCard.jsx
import { useState } from 'react';

const ProfileCard = ({ name, bio, photo }) => {
  // Three independent state variables
  const [isFollowing, setIsFollowing] = useState(false);
  const [likeCount, setLikeCount]   = useState(0);
  const [isExpanded, setIsExpanded] = useState(false);

  return (
    <div className="profile-card">
      <img src={photo} alt={name} />
      <h3>{name}</h3>

      {isExpanded && <p>{bio}</p>}

      <button onClick={() => setIsExpanded(!isExpanded)}>
        {isExpanded ? 'Show Less' : 'Read More'}
      </button>

      <button onClick={() => setIsFollowing(!isFollowing)}>
        {isFollowing ? '✅ Following' : 'Follow'}
      </button>

      <button onClick={() => setLikeCount(likeCount + 1)}>
        ❤️ {likeCount}
      </button>
    </div>
  );
};

Each state is independent

  • Clicking Follow doesn't affect likeCount
  • Clicking ❤️ doesn't collapse the bio
  • React knows which piece of state changed
  • Only the affected part of the UI re-renders

The Toggle Pattern

// Toggle = flip current value to opposite
setIsFollowing(!isFollowing);
setIsExpanded(prev => !prev); // safer form

Pro tip: If two state variables always change together, consider merging them into one object. If they change independently, keep them separate.

Event Handling in React

React wraps native browser events in a Synthetic Event object. The API is almost identical to vanilla JS — but it's JSX attributes on your elements, not addEventListener.

❌ Vanilla JS (imperative)

// JavaScript:
document
  .querySelector('button')
  .addEventListener('click', handleClick);

// And you have to remember to remove it!
button.removeEventListener('click', handleClick);

✅ React (declarative)

// JSX — event handlers as props:
<button onClick={handleClick}>Click me</button>
// React attaches and cleans up automatically ✅

Common React Event Props

onClick — button, div, any element
onChange — input, textarea, select
onSubmit — form element
onFocus — when input gains focus
onBlur — when input loses focus
onMouseEnter — hover start
onKeyDown — key pressed

⚠️ Pass the function, don't call it:
❌ onClick={handleClick()} — runs immediately!
✅ onClick={handleClick} — runs on click

Event Handling

The Event Object — Reading What the User Did

// React automatically passes the event object
const handleChange = (e) => {
  console.log(e.target.value);  // current input text
  console.log(e.target.name);   // input's name attr
  console.log(e.target.checked); // for checkboxes
};

const handleClick = (e) => {
  e.preventDefault(); // stop default browser action
  console.log(e.target); // the clicked element
};

const handleSubmit = (e) => {
  e.preventDefault(); // ALWAYS for forms!
  // Now handle the form data yourself
};

// Wire them up:
<input onChange={handleChange} />
<button onClick={handleClick}>Go</button>
<form onSubmit={handleSubmit}>...</form>

The Most Important Property: e.target.value

For any text input or textarea, e.target.value gives you the current text the user has typed. This is what you pass to your state setter to keep React in sync with the input.

⚠️ Always preventDefault on forms

Without it, submitting a form reloads the entire page — wiping all your React state. e.preventDefault() stops the browser's default form submission behaviour.

Inline with the event object

// Also works inline (e is auto-passed):
<input
  onChange={(e) => setName(e.target.value)}
/>

Controlled Components — React Owns the Input

❌ Uncontrolled — DOM owns the value

// No value prop — DOM tracks the text
<input type="text" />

const [email, setEmail] = useState('');

<input
  type="email"
  value={email}             ← state drives the display
  onChange={(e) => setEmail(e.target.value)}

// React state = single source of truth ✅

User types "T" in the input

↓
2

onChange fires → setEmail("T")

↓
3

React re-renders with email = "T"

↓
4

Input shows value="T" from state

↓
5

Repeat for every keystroke. State = truth ✅

Before vs After — Making an Input Controlled

❌ Before — Uncontrolled input

const SearchBox = () => (
  <div>

      type="text"

Common Mistakes

Direct Mutation vs Setter — The Most Common Bug

❌ WRONG — Direct Mutation

// ❌ NEVER modify state directly

// Primitive — looks like it works but doesn't:
count = count + 1;         // ❌
count++;                   // ❌

// Array — mutating breaks everything:
items.push(newItem);      // ❌ React can't see this
items[0] = 'Updated';    // ❌ Never index-assign

// Object — mutating properties:
user.name = 'Tolu';       // ❌
user.city = 'Lagos';     // ❌

Why it breaks: React detects changes by comparing the old and new values by reference. Mutating the same object/array means the reference never changes. React thinks nothing happened and skips the re-render.

✅ RIGHT — Always use the Setter

// ✅ Always use the setter function

// Primitive — just pass the new value:
setCount(count + 1);      // ✅
setCount(prev => prev + 1); // ✅ safer

// Array — create a NEW array:
setItems([...items, newItem]); // ✅ spread
setItems(items.filter(i => i.id !== id)); // ✅

// Object — create a NEW object:
setUser({ ...user, name: 'Tolu' });  // ✅
setUser(prev => ({ ...prev, city: 'Lagos' })); // ✅

Golden rule: State must be replaced, not modified. Always give the setter a brand-new value. Use spread (...) for arrays and objects to copy the old data and override what changed.

Common State Pitfalls — And How to Fix Them

🐛 Pitfall 1 — Stale state after setter

setCount(count + 1);
console.log(count); // Still old value!
// Fix: use the new value directly, or
// rely on the re-render to see it.

🐛 Pitfall 2 — Calling setState in the render body

// ❌ This causes an infinite loop!
const App = () => {
  const [n, setN] = useState(0);
  setN(n + 1); // ← runs every render = infinite
  return <p>{n}</p>;
};

🐛 Pitfall 3 — Forgetting to import useState

// ❌ Error: "useState is not defined"
// Fix — add to your import:
import { useState } from 'react'; // ✅

🐛 Pitfall 4 — Missing value prop on controlled input

// ❌ onChange without value = broken state
<input onChange={(e) => setName(e.target.value)} />
// ✅ Both value AND onChange required:
<input value={name} onChange={(e) => setName(e.target.value)} />

🐛 Pitfall 5 — Calling the handler instead of passing it

// ❌ Calls immediately on render, not on click:
<button onClick={handleClick()}>...</button>
// ✅ Pass the function reference:
<button onClick={handleClick}>...</button>

Quick Debug Checklist

  • Did you import useState?
  • Is the setter called inside an event handler?
  • Does the input have both value and onChange?
  • Are you reading state, not mutating it?

Live Coding — Like Button with Internal State

Live Coding 
// src/components/LikeButton.jsx
import { useState } from 'react';

const LikeButton = ({ initialCount = 0 }) => {
  const [liked, setLiked]   = useState(false);
  const [count, setCount]   = useState(initialCount);

  const handleLike = () => {
    if (liked) {
      setLiked(false);
      setCount(prev => prev - 1);
    } else {
      setLiked(true);
      setCount(prev => prev + 1);
    }
  };

  return (
    <button
      onClick={handleLike}
      className={`like-btn ${liked ? 'liked' : ''}`}
    >
      {liked ? '❤️' : '🤍'} {count}
    </button>
  );
};

export default LikeButton;

// Use it anywhere:
<LikeButton initialCount={42} />
<LikeButton initialCount={128} />

Design Patterns at Work:

  • Props + State together: initialCount from parent, liked managed internally
  • Derived className: CSS class changes based on liked
  • Two states, one handler: both update in one click
  • Functional updates: prev => prev + 1 is safe
  • Emoji as UI: ❤️ / 🤍 is an instant visual indicator

Each instance has its own state

Every <LikeButton /> you render has independent state. Liking one doesn't affect the others. This is the power of component encapsulation.

Forms — Full Pattern: Input → State → Display

Code Pattern 
// src/components/GreetingForm.jsx
import { useState } from 'react';

const GreetingForm = () => {
  const [name, setName] = useState('');

  return (
    <div className="greeting-form">

      {/* 1. Live display — updates as you type */}
      <h2>
        {name ? `Hello, ${name}! 👋` : 'What is your name?'}
      </h2>

      {/* 2. Controlled input */}
      <input
        type="text"
        value={name}
        onChange={(e) => setName(e.target.value)}
        placeholder="Type your name..."
        maxLength={30}
      />

      {/* 3. Live character count */}
      <p className="char-count">
        {name.length} / 30
      </p>

      {/* 4. Clear button */}
      <button onClick={() => setName('')}>
        Clear
      </button>
    </div>
  );
};

✅ What's happening here:

  • name starts empty ('')
  • Every keystroke updates name via setName
  • The greeting at the top re-renders live with the new value
  • Character count is just name.length — no extra state needed
  • Clear resets state to '' — no DOM touching

Derived values = no extra state needed

name.length, name.toUpperCase(), name.split(' ').length — these are all derived from state. Compute them inline. Never create a separate state variable for something you can derive.

Lifting State Up — When Two Components Need It

When two sibling components need the same state, lift it to their nearest common parent. Pass the value and setter down as props.

❌ State trapped in wrong component

// Input component owns the value,
// but Display component can't see it!
<Input />     ← has state
<Display />  ← needs state but can't reach it

✅ State lifted to parent

const App = () => {
  const [value, setValue] = useState('');
  return (
    <>
      <Input value={value} onChange={setValue} />
      <Display value={value} />  ← now it can see it
    </>
  );
};

The Rule: Colocation vs Lifting

  • Keep state where it's used. Don't lift unnecessarily.
  • Lift when two or more components need the same data.
  • Lift to the nearest common ancestor — not to App unless you must.
  • The Celsius/Fahrenheit converter in your homework uses exactly this pattern.

Coming in Class 16: Context

When prop-drilling gets too deep, useContext lets you share state across the entire tree without passing props through every layer. But first, master lifting state — it's the foundation.

      value={p}
      checked={plan === p}
      onChange={(e) => setPlan(e.target.value)}
    />
    {p}
  </label>
))}

The Key Difference

Text inputs: use value + e.target.value
Checkboxes: use checked + e.target.checked
Selects: use value + e.target.value
Radios: use checked={state === option}

What's Next — Class 15: useEffect & Side Effects

You've mastered the core loop

State → Event → Setter → Re-render

This loop drives 90% of everything you'll build. Class 15 adds the missing piece: what happens outside the render cycle.

What useEffect does:

  • Fetch data from an API when the component mounts
  • Run code when specific state or props change
  • Set up and clean up timers, subscriptions, listeners
  • Sync React state with external systems
// Preview — Class 15 useEffect
import { useState, useEffect } from 'react';

const UserProfile = ({ userId }) => {
  const [user, setUser] = useState(null);

  useEffect(() => {
    // Runs after the component renders
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => setUser(data));
  }, [userId]); // ← runs when userId changes

  if (!user) return <p>Loading...</p>;

  return <h2>{user.name}</h2>;
};

After Class 15, your components will be able to load real data from APIs, sync with external services, and come fully alive as production-ready applications.

Class 14 — What You Now Know

useState Hook

  • Import from 'react'
  • Returns [value, setter]
  • Any initial data type
  • Setter triggers re-render
  • Top-level only (Rules of Hooks)

Event Handling

  • camelCase event props in JSX
  • Pass the function, don't call it
  • e.target.value for inputs
  • e.preventDefault() for forms
  • e.target.checked for checkboxes

Controlled Forms

  • value prop drives display
  • onChange updates state
  • React = single source of truth
  • Reset = set state to default
  • Derive values, don't duplicate state

The Shift in Thinking 🧠

You no longer manipulate the DOM. You manage state. You declare what the UI should look like for a given state, and let React handle every update. This is the mental model of every professional React developer.
