What Is Routing? SPAs vs Traditional Sites

🧠 In Plain English

Old websites: click a link → browser goes to the server → server sends a whole new HTML page → browser reloads. Like driving to a new location every time.

React (SPA): the browser downloads the whole app once. When you click a link, React swaps out the content instantly — no server trip, no reload. Like changing rooms inside a building instead of leaving and entering a new one.

Traditional Website vs SPA

Traditional: click /about → full page reload

Browser → Server → New HTML → Flash → Reload ❌

SPA: click /about → instant swap

React swaps the component. No server trip. No flash. ✅

Apps you use every day that use this:

Gmail, Instagram, Twitter/X, YouTube, LinkedIn, Airbnb — all SPAs. The URL changes but there is no page reload.

Installing React Router — 2 Steps

Step 1 — Install the package

# In your terminal, inside your project folder:
npm install react-router-dom

Common mistake: typing react-router instead of react-router-dom. The -dom part is required for browser apps.

Step 2 — Wrap your app in BrowserRouter

// src/main.jsx — add BrowserRouter here
import { BrowserRouter } from 'react-router-dom';
import App from './App';

ReactDOM.createRoot(document.getElementById('root'))
  .render(
    <BrowserRouter>   ← wrap App here
      <App />
    </BrowserRouter>
  );

🧠 Why Does BrowserRouter Go in main.jsx?

Think of BrowserRouter as a power socket. Every component that needs routing must be plugged into it. By wrapping the entire app, every component anywhere in the tree can access routing. If you put it too deep, only part of the app gets routing.

✅ Checklist after install

  • react-router-dom appears in package.json
  • <BrowserRouter> wraps <App /> in main.jsx
  • No errors in the console
  • App still loads normally

🤖 If you get an error after installing: paste the error into Claude with "I just installed react-router-dom in my Vite React app. I'm getting this error: [error]"

Routes & Route — Your App's Menu of Pages

🧠 In Plain English

Routes is like a menu at a restaurant. Route is each item on the menu. When the URL matches a route's path, React shows that route's element. Only the matching one. If nothing matches, you fall through to your 404 page.

// src/App.jsx — this is your route map
import { Routes, Route } from 'react-router-dom';
import HomePage    from './pages/HomePage';
import AboutPage   from './pages/AboutPage';
import ProductPage from './pages/ProductPage';
import NotFound    from './pages/NotFound';
import Navbar      from './components/Navbar';

const App = () => (
  <div>
    <Navbar />  ← shows on EVERY page

    <Routes>
      <Route path="/"           element={<HomePage />} />
      <Route path="/about"      element={<AboutPage />} />
      <Route path="/products/:id" element={<ProductPage/>} />
      {/* :id is a variable — matches /products/1, /products/42 */}
      <Route path="*"           element={<NotFound />} />
      {/* * matches EVERYTHING else — your 404 */}
    </Routes>
  </div>
);
export default App;

How Paths Map to URLs

/   → your-app.com/
/about → your-app.com/about
/products/:id → your-app.com/products/42
*   → everything else → 404

Folder Structure Best Practice

src/
├── pages/         ← full page components
│   ├── HomePage.jsx
│   ├── AboutPage.jsx
│   └── ProductPage.jsx
├── components/    ← reusable pieces
│   ├── Navbar.jsx
│   └── ProductCard.jsx
└── App.jsx        ← the route map

Link & NavLink — Never Use <a> Again

🧠 In Plain English

In normal HTML you use <a href="/about"> to link pages. The problem: it causes a full page reload, losing all your React state.

React Router's <Link> looks and works like <a> but uses JavaScript to switch pages instantly — no reload, no lost state.

❌ Wrong — reloads the page

<a href="/about">About</a>
// This triggers a full browser reload
// All your React state is gone! ❌

✅ Right — instant navigation

import { Link } from 'react-router-dom';

<Link to="/about">About</Link>
// No reload. React swaps the page. ✅

NavLink — Adds "active" class automatically

import { NavLink } from 'react-router-dom';

// Use in your Navbar component:
const Navbar = () => (
  <nav>
    <NavLink
      to="/"
      style={({ isActive }) => ({
        color: isActive ? '#6366f1' : '#a0aec0',
        fontWeight: isActive ? '700' : '400'
      })}
    >Home</NavLink>

    <NavLink to="/about" /* same style */>About</NavLink>
    <NavLink to="/products">Products</NavLink>
  </nav>
);

NavLink knows which page you're on. Use it for navbars where the active link should look different. Link is for any other navigation (buttons, cards, etc.)

useParams — Dynamic Pages Like /products/42

Live Coding

🧠 In Plain English

Twitter has millions of users but ONE user profile page component. The URL changes — /user/tolu, /user/emeka — and the same component uses the name from the URL to show different data.

useParams() reads that variable part from the URL and gives it to your component.

// App.jsx — the :id is the "variable slot"
<Route path="/users/:id" element={<UserPage/>} />

// UserPage.jsx — reads the :id from the URL
import { useParams, useNavigate } from 'react-router-dom';
import { useState, useEffect } from 'react';

const UserPage = () => {
  const { id } = useParams();
  // id = "tolu" if URL is /users/tolu
  // id = "42"   if URL is /users/42

  const [user, setUser]     = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // Fetch data for THIS specific user
    fetch(`https://randomuser.me/api/?seed=${id}`)
      .then(r => r.json())
      .then(d => { setUser(d.results[0]); setLoading(false); });
  }, [id]); // re-run when :id changes

  if (loading) return <p>Loading...</p>;
  return <h2>{user?.name.first}</h2>;
};

The useParams Pattern

  1. Define :param in the Route path
  2. Call useParams() in the page component
  3. Destructure the param: const { id } = useParams()
  4. Use it in a fetch URL or to filter your data
  5. Put [id] in useEffect's dependency array

Real uses of dynamic routes

/products/:id
/movies/:movieId
/users/:username
/countries/:code

useNavigate & 404 Pages

useNavigate — Go to a page from code

🧠 In Plain English

Link works for clicking. useNavigate is for going somewhere after something happens — like after a form submits successfully, or after an item is deleted. "Take the user to the home page after they log out."

import { useNavigate } from 'react-router-dom';

const LoginPage = () => {
  const navigate = useNavigate();

  const handleLogin = () => {
    // ... check credentials ...
    navigate('/dashboard'); // go here after success
  };

  // Go back to the previous page:
  const goBack = () => navigate(-1);

  return (
    <div>
      <button onClick={goBack}>← Back</button>
      <button onClick={handleLogin}>Login</button>
    </div>
  );
};

404 Page — The Catch-All

// pages/NotFound.jsx
import { Link } from 'react-router-dom';

const NotFound = () => (
  <div style={{textAlign: 'center', padding: '50px'}}>
    <h1>404</h1>
    <p>This page doesn't exist.</p>
    <Link to="/">← Back to Home</Link>
  </div>
);
export default NotFound;
// In App.jsx — the * path catches everything
<Route path="*" element={<NotFound/>} />
// Must be the LAST route

navigate(-1) is like the browser's back button. It goes to wherever the user was before. Perfect for "Back" buttons on detail pages.

Full Example — Country Explorer App

Live Coding 
// App.jsx — 3-page country explorer
import { Routes, Route } from 'react-router-dom';
import Navbar       from './components/Navbar';
import HomePage    from './pages/HomePage';
import CountryPage from './pages/CountryPage';
import NotFound    from './pages/NotFound';

const App = () => (
  <div>
    <Navbar />
    <Routes>
      <Route path="/"             element={<HomePage/>}    />
      <Route path="/country/:name" element={<CountryPage/>} />
      <Route path="*"             element={<NotFound/>}    />
    </Routes>
  </div>
);

// HomePage.jsx — list of countries as links
const HomePage = () => {
  const [countries, setCountries] = useState([]);
  useEffect(() => {
    fetch('https://restcountries.com/v3.1/region/africa?fields=name,flags')
      .then(r => r.json()).then(setCountries);
  }, []);

  return (
    <div>
      {countries.map(c => (
        <Link key={c.name.common}
              to={`/country/${c.name.common}`}>
          {c.name.common}
        </Link>
      ))}
    </div>
  );
};
// CountryPage.jsx — detail page
import { useParams, useNavigate } from 'react-router-dom';

const CountryPage = () => {
  const { name }  = useParams();
  const navigate   = useNavigate();
  const [country, setCountry] = useState(null);

  useEffect(() => {
    fetch(`https://restcountries.com/v3.1/name/${name}`)
      .then(r => r.json())
      .then(data => setCountry(data[0]));
  }, [name]);

  if (!country) return <p>Loading...</p>;

  return (
    <div>
      <button onClick={() => navigate(-1)}>← Back</button>
      <img src={country.flags.svg} width="120" />
      <h1>{country.name.common}</h1>
      <p>Capital: {country.capital?.[0]}</p>
      <p>Pop: {country.population.toLocaleString()}</p>
    </div>
  );
};

What we just built: A list page → click → detail page → back button. This exact pattern is how every e-commerce, movie, and news app works.

Component Structure — The Rules That Save You

🧠 In Plain English

A good component is like a good function: it does ONE thing, does it well, and doesn't care about things outside its job. When you open a component file, you should immediately know what it does from its name alone.

✅ Signs of a well-structured component

  • File name = component name (MovieCard.jsx)
  • Does one job (shows a movie card)
  • Under ~100 lines. If longer, split it
  • Receives data via props, doesn't fetch its own
  • Could be reused in multiple places

❌ Warning signs to fix

  • Component doing fetch + render + user interaction
  • One JSX file that's 300+ lines
  • Copying the same JSX block in multiple files
  • Passing props down 3+ levels (prop drilling)

Recommended Folder Structure

src/
├── pages/           ← full-page components
│   ├── HomePage.jsx
│   ├── MoviePage.jsx
│   └── NotFound.jsx
│
├── components/      ← reusable smaller pieces
│   ├── Navbar.jsx
│   ├── MovieCard.jsx
│   ├── SearchBar.jsx
│   └── LoadingSpinner.jsx
│
├── hooks/           ← custom hooks (advanced)
│   └── useMovies.js
│
├── App.jsx          ← route definitions only
└── main.jsx         ← BrowserRouter lives here

State Strategy — Where Should State Live?

🧠 The Golden Rule of State

State should live in the lowest component that needs it — but high enough so all components that need it can reach it. If only MovieCard needs its own hover state — keep it there. If the search query is needed by three different components — lift it up to their shared parent.

Local State — keep it here

  • Hover / focus effects
  • Open/close of a modal
  • Show/hide a section
  • Input value (typed text)

Only used in that one component

Lifted State — parent holds it

  • Search query (affects results)
  • Active filter (affects the list)
  • Fetched movies list
  • Selected item

Multiple children need the same data

What NOT to put in state

  • Values you can calculate from state
  • Duplicate data (store once)
  • Data that never changes
// Don't store computed values:
// const [count, setCount] = useState(0);
// const [doubled, setDoubled] = ... ← NO
// const doubled = count * 2; ← YES

Common React Router Bugs & Fixes

Bug: "useParams must be used within a Router"

Cause: BrowserRouter is missing or in the wrong place.

Fix: Go to main.jsx. Wrap <App /> with <BrowserRouter>.

Bug: Clicking a link reloads the whole page

Cause: You're using <a href> instead of <Link to>.

Fix: Replace every <a href="..."> with <Link to="..."> and import Link from 'react-router-dom'.

Bug: Detail page shows the right URL but is blank

Cause: The Route path doesn't match the Link's to value, OR useParams name doesn't match the :param in path.

Fix: Make sure path="/movie/:id" and to={`/movie/${movie.id}`} and const {'{id}'} = useParams() all use the same name.

Bug: "Cannot read properties of undefined (reading 'name')"

Cause: The API data hasn't loaded yet, but you're already trying to access fields on it.

Fix: Add early returns. if (!movie) return <p>Loading...</p> before accessing movie.name.

Bug: 404 page shows up even on valid routes

Cause: The path="*" route is listed before your real routes.

Fix: Always put path="*" as the very LAST Route inside <Routes>.

🤖 For any bug: Paste the full error message + your App.jsx and the relevant component into Claude. Say what page you're on and what you expected to happen.
