Rendering a List — The .map() Pattern

🧠 In Plain English

You have an array of items (names, products, users, jokes). You want to show each one on screen. Instead of writing the same JSX 10 times, you use .map() to turn the array into JSX automatically.

❌ Writing it manually (the painful way)

// You have 10 foods. Do you write this 10 times?
<p>Jollof Rice</p>
<p>Suya</p>
<p>Puff Puff</p>
<p>Egusi Soup</p>
// ... 6 more lines? NO! ❌

✅ Using .map() — write it ONCE

const foods = ['Jollof Rice', 'Suya', 'Puff Puff', 'Egusi'];

const FoodList = () => (
  <ul>
    {foods.map(food => (
      <li key={food}>{food}</li>
    ))}
  </ul>
);

Add 100 more foods to the array. The JSX doesn't change at all. React renders all of them automatically. ✅

⚠️ See that key={food}? That's required. Every item in a .map() list needs a unique key. Next slide explains why.

The key Prop — Why React Needs It

🧠 In Plain English

Imagine you have 5 identical-looking boxes on a conveyor belt. How do you tell them apart? You label them: Box 1, Box 2, Box 3...

React does the same thing with your list items. The key is the label. It helps React know which item changed when your list updates.

❌ Wrong — using array index

{items.map((item, index) => (
  <li key={index}>{item}</li>
))}
// If list order changes, keys change too. Bugs!

✅ Right — use a unique ID

const users = [
  { id: 1, name: 'Tolu' },
  { id: 2, name: 'Emeka' },
];

{users.map(user => (
  <li key={user.id}>{user.name}</li>
))}

The 3 Rules of Keys

  • Must be unique among siblings
  • Use IDs from your data, not array index
  • Goes on the outermost element in .map()

Missing key = yellow warning in console. React will still work, but slower and with potential bugs on re-renders.

const students = [
  { id: 1, name: 'Chiamaka', score: 74, city: 'Onitsha'  },
  { id: 2, name: 'Bayo',     score: 88, city: 'Ibadan'  },
  { id: 3, name: 'Ngozi',    score: 55, city: 'Enugu'   },
  { id: 4, name: 'Tunde',    score: 92, city: 'Lagos'   },
];

const StudentList = () => (
  <div className="student-list">
    <h2>Students</h2>
    {students.map(student => (
      <div key={student.id} className="student-card">
        <h3>{student.name}</h3>
        <p>Score: {student.score}</p>
        <p>City: {student.city}</p>
      </div>
    ))}
  </div>
);

The Pattern — memorise this:

1. Start with your data array
2. Call .map() on it
3. Return JSX for each item
4. Add key={item.id}

Combine with spread operator

// You can also spread all props at once:
{students.map(s => (
  <StudentCard
    key={s.id}
    {...s}   ← passes all fields as props
  />
))}

Add a 5th student to the array. The page updates automatically. No JSX changes. This is why React exists.

Showing & Hiding Things — 3 Simple Patterns

🧠 In Plain English

Sometimes you want to show a "Welcome back!" message only if the user is logged in. Or show "Out of Stock" only when an item has no stock. You only want to render something when a condition is true. These three patterns cover 95% of cases.

Pattern 1: && (show OR nothing)

// Only shows if isLoggedIn is true
{isLoggedIn &&
  <p>Welcome back! 👋</p>
}

// Only shows badge if isPassing
{score >= 60 &&
  <span>✅ Passed</span>
}

Use when: show something OR show nothing.

Pattern 2: Ternary (A or B)

// Shows one OR the other
<p>
  {isLoggedIn
    ? 'Welcome back!'
    : 'Please log in'
  }
</p>

// Button text changes:
<button>
  {liked ? '❤️ Liked' : '🤍 Like'}
</button>

Use when: show A if true, B if false.

Pattern 3: Early Return

// Return something completely different
const Profile = ({ user }) => {

  // If no user, bail out early
  if (!user) return (
    <p>Loading...</p>
  );

  // Otherwise show the real profile
  return <h2>{user.name}</h2>;
};

Use when: the whole component changes.

Live Build — Todo List (Lists + State Together)

Live Coding 
import { useState } from 'react';

const TodoList = () => {
  const [todos, setTodos]   = useState([]);
  const [input, setInput]   = useState('');

  // Add a new todo
  const addTodo = () => {
    if (!input.trim()) return; // ignore empty
    setTodos(prev => [
      ...prev,
      { id: Date.now(), text: input, done: false }
    ]);
    setInput(''); // clear input
  };

  // Toggle done/not done
  const toggleTodo = (id) => {
    setTodos(prev => prev.map(t =>
      t.id === id ? { ...t, done: !t.done } : t
    ));
  };

  // Delete a todo
  const deleteTodo = (id) => {
    setTodos(prev => prev.filter(t => t.id !== id));
  };

  return (
    <div>
      <input value={input}
             onChange={e => setInput(e.target.value)}
             placeholder="Add a task..." />
      <button onClick={addTodo}>Add</button>

      {todos.map(todo => (
        <div key={todo.id}>
          <span style={{
            textDecoration: todo.done ? 'line-through' : 'none'
          }}>{todo.text}</span>
          <button onClick={() => toggleTodo(todo.id)}>✓</button>
          <button onClick={() => deleteTodo(todo.id)}>✕</button>
        </div>
      ))}

      {/* Conditional: show count only if todos exist */}
      {todos.length > 0 &&
        <p>{todos.filter(t => !t.done).length} remaining</p>
      }
    </div>
  );
};

🧠 What's happening here

  • todos is an array in state
  • input is the controlled text box
  • Add: spread old todos + new one
  • Toggle: .map() through, flip done
  • Delete: .filter() removes by id
  • Display: .map() renders the list

3 concepts working together:

useState (from Class 14) → stores the list
.map() → renders each item
Conditional → shows count only when needed

useEffect — "Do This When Something Happens"

🧠 In Plain English

Think of useEffect like telling your phone: "When I open WhatsApp, download my new messages."

In React: "When this component appears on screen, go fetch some data."

It runs code after the component renders — not during. This keeps React fast.

import { useState, useEffect } from 'react';

const MyComponent = () => {

  useEffect(() => {
    // This code runs AFTER the component renders
    console.log('Component appeared!');

  }, []); // ← the [] means: run ONCE on mount

  return <p>Hello</p>;
};

The Dependency Array — The Most Important Part

useEffect(() => {}, [])
↑ Empty [] = run once when component loads
useEffect(() => {}, [userId])
↑ [userId] = run when userId changes
useEffect(() => {})
↑ No [] = runs on EVERY render (usually wrong!)

⚠️ Forgetting the [] is the most common mistake. Without it, your useEffect runs on every single render — which usually causes an infinite loop.

Fetching Data — The Standard Pattern

🧠 In Plain English

An API is like a waiter. You say "I want a random joke", the waiter goes to the kitchen (the server), gets it, and brings it back. This takes a moment — so while you wait, you show a loading spinner. If the waiter spills the food, you show an error message.

import { useState, useEffect } from 'react';

const JokeApp = () => {
  const [joke, setJoke]       = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError]     = useState(null);

  const fetchJoke = async () => {
    setLoading(true);
    setError(null);
    try {
      const res  = await fetch(
        'https://official-joke-api.appspot.com/random_joke'
      );
      const data = await res.json();
      setJoke(data);
    } catch (err) {
      setError('Could not load joke 😢');
    } finally {
      setLoading(false);
    }
  };

  // Fetch when component loads
  useEffect(() => { fetchJoke(); }, []);

  if (loading) return <p>Loading...</p>;
  if (error)   return <p>{error}</p>;

  return (
    <div>
      <p>{joke.setup}</p>
      <p>{joke.punchline}</p>
      <button onClick={fetchJoke}>Next Joke</button>
    </div>
  );
};

The 3 States Every API Call Needs

1

loading — show spinner while waiting

2

error — show message if it fails

3

data — show content when it works

This pattern works for every API call you'll ever write. Change the URL, change what you display — everything else stays the same.

Free APIs to practice with:
Jokes: official-joke-api.appspot.com
Users: randomuser.me/api
Quotes: quotable.io/random

Fetching a List — .map() Meets useEffect

Live Coding 
// Fetch 5 random users and display them as cards
import { useState, useEffect } from 'react';

const UserFeed = () => {
  const [users, setUsers]     = useState([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch('https://randomuser.me/api/?results=5')
      .then(res => res.json())
      .then(data => {
        setUsers(data.results);
        setLoading(false);
      });
  }, []); // ← run once

  if (loading) return <p>Loading users...</p>;

  return (
    <div className="user-feed">
      {users.map(user => (
        <div key={user.login.uuid} className="user-card">
          <img src={user.picture.medium} alt="user" />
          <h3>{user.name.first} {user.name.last}</h3>
          <p>{user.email}</p>
        </div>
      ))}
    </div>
  );
};

What just happened — step by step:

  1. Component renders, shows "Loading users..."
  2. useEffect fires after render
  3. fetch() calls the randomuser.me API
  4. API returns 5 user objects as JSON
  5. setUsers() stores them in state
  6. setLoading(false) triggers re-render
  7. .map() turns each user into a card
  8. 5 cards appear on screen ✅

Reload vs Refetch

To get new users without page reload, just call fetchUsers() from a button's onClick. React refetches and re-renders the new list.

Loading & Error States — Don't Leave Users Confused

🧠 In Plain English

Ever open an app and it just shows a blank white screen? That's bad UX. Always show the user what's happening — whether it's loading, broken, or empty.

const DataComponent = () => {
  const [data, setData]       = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError]     = useState(null);

  // ... fetch logic in useEffect

  // Handle each state with early returns:
  if (loading) return (
    <div className="loader">⏳ Loading...</div>
  );

  if (error) return (
    <div className="error">
      ❌ {error}
      <button onClick={fetchData}>Try again</button>
    </div>
  );

  if (data.length === 0) return (
    <div>📭 No data found.</div>
  );

  // Only reach here if everything worked:
  return <div>{data.map(...)}</div>;
};

The 4 States Every Data Component Has

⏳

Loading

Show a spinner or "Loading..." text

❌

Error

Show what went wrong + a retry button

📭

Empty

Show "No results found" — not a blank page

✅

Success

Render the actual data

Professional apps always handle all four. Check any real app — Twitter, Instagram, Google — they never show a blank screen. Neither should yours.

useEffect with Dependencies — Re-fetch When Things Change

🧠 In Plain English

Imagine you're searching for products. Every time you change what you're searching for, the app should go fetch new results. That's what the dependency array does — it re-runs the effect whenever the listed value changes.

const ProductSearch = () => {
  const [query, setQuery]       = useState('phone');
  const [results, setResults]   = useState([]);
  const [loading, setLoading]   = useState(false);

  // Re-runs every time 'query' changes
  useEffect(() => {
    setLoading(true);
    fetch(`https://fakestoreapi.com/products?q=${query}`)
      .then(r => r.json())
      .then(data => {
        setResults(data);
        setLoading(false);
      });
  }, [query]); // ← re-fetch when query changes

  return (
    <div>
      <input
        value={query}
        onChange={e => setQuery(e.target.value)}
        placeholder="Search..."
      />
      {loading && <p>Searching...</p>}
      {results.map(p => (
        <p key={p.id}>{p.title}</p>
      ))}
    </div>
  );
};

Dependency Array Cheat Sheet

[]     Run once when component mounts
[x]   Run when x changes
[x,y] Run when x OR y changes
none  Run every render (bad!)

⚠️ Common Mistake: Infinite Loop

// ❌ This loops forever!
useEffect(() => {
  setData(something); // updates state
}, [data]);             // watches state → loops!

// Fix: remove data from the dependency array

How to Ask AI for Help — Prompting That Works

The quality of the answer depends entirely on the quality of the question. Here's the difference:

❌ Bad prompts (get vague answers)

You typed:
"my code doesn't work fix it"

"how do I make react work"

"useEffect help"

AI has no context. It gives a generic answer that doesn't match your problem. You're going in circles.

✅ Good prompts (get precise fixes)

You typed:
"I'm building a React app with Vite. I have a useEffect that fetches from an API, but it's running in an infinite loop. Here's my code: [paste code]. What's causing the loop and how do I fix it?"

AI has all the context. It identifies your dependency array mistake and gives you the exact fix with an explanation.

The 4-Part Prompt Formula

Context (I'm using React + Vite)  +  What I'm trying to do (fetch user data)  +  What's going wrong (infinite loop)  +  My code (paste it)

Debugging with AI — Paste the Error, Get the Fix

Live Demo

You see this error in the console:

TypeError: Cannot read properties of undefined
(reading 'map')

at StudentList (StudentList.jsx:12)
at renderWithHooks
...

This used to mean: 30 minutes of Googling. Now it means: 30 seconds with AI.

What to paste into Claude or ChatGPT:
"I'm getting this error in my React component: [paste the full error]. Here's my component code: [paste the code]. What's causing it and how do I fix it?"

What AI will tell you:

The error "Cannot read properties of undefined (reading 'map')" means you're calling .map() on something that isn't an array — it's undefined. This usually happens because:

1. Your API response has a different structure than expected
2. Your state starts as null instead of []

Fix: Change useState(null) to useState([]), or add a guard: data?.map(...)

✅ Ask AI to explain, not just fix

Add "explain why this happened" to your prompt. Understanding the error means you won't make the same mistake twice.

Using AI to Learn — Not Just to Copy

Ask AI to explain code you don't understand

Prompt:
"Explain this React code to me line by line, as if I'm a beginner: [paste code]"

Ask AI to review your code

Prompt:
"Review this React component. Tell me: is there anything wrong? Are there better ways to write this? Am I handling errors correctly? [paste code]"

Ask AI to simplify something confusing

Prompt:
"I don't understand the dependency array in useEffect. Explain it using a simple, everyday analogy. No jargon."

What NOT to do with AI

  • ❌ Ask AI to write your entire homework
  • ❌ Copy AI code without reading it
  • ❌ Submit AI code as your understanding
  • ❌ Trust AI blindly — it makes mistakes

✅ The Smart Way to Use AI

  • Try the problem yourself first — even 10 mins
  • Ask AI to explain the concept, then write the code
  • Use AI code as a model, then write your own version
  • After AI gives an answer, ask: "why does this work?"

In your capstone project: interviewers will ask you to explain your code. If AI wrote it and you don't understand it, you'll fail the interview. Use AI to learn faster — not to skip learning.
